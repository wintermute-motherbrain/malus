# MetalContext is single-consumer by design; not thread-safe under concurrent host access

## Decision

`malus-runtime`'s process-global `static CONTEXT: OnceLock<MetalContext>` (`metal.rs:81`) is designed to be used by one host OS thread at a time. This is not merely undocumented — it's an explicit choice: kernel ids allocated by `compile_kernels()` are unique only *within* one compilation, and the shared `current_command_buffer`/`pipelines` slots assume no two unrelated programs are mid-flight concurrently. We are not making `MetalContext` safe under concurrent host-thread access.

Instead:
- Kernel ids are made globally unique *within a process* via a process-global `AtomicU64` counter in `compile_kernels()`, so sequential (non-concurrent) compilations in one process never collide (extends the "collision-free by construction" principle from ADR-0010, just widened from per-compilation to per-process scope). This is defense-in-depth, not load-bearing for the hang fix below: once test execution is serialized, no two compilations ever race for the same id regardless of numbering scheme.
- Test code that exercises the real Metal backend (`malus-runtime/src/tests.rs`, `malus-codegen-cpu/tests/metal_integration.rs`) is required to serialize access via a `Mutex<()>` test lock, held for the duration of each test. Cargo's default multi-threaded test harness otherwise runs these tests concurrently against the one shared `CONTEXT`, which risks intermittent GPU-side `barrier()` deadlocks (wrong pipeline bound to a colliding kernel id → not all threads in a threadgroup reach the barrier → `waitUntilCompleted()` hangs forever, with no timeout).

### Postscript: the actual hang was a self-deadlock, not the missing lock

Diagnosis started from `malus-runtime/src/tests.rs` having zero synchronization (79 tests, no lock) while `metal_integration.rs` already had one. Verification after applying the fixes above still hung. Root cause turned out to be a pre-existing bug in `metal_integration.rs`, independent of the missing-lock gap: `test_gradient_check_all_ops` acquired `METAL_TEST_LOCK` directly, then called `run_metal_src()`, which *also* acquired the same non-reentrant `Mutex` — a guaranteed self-deadlock on every run, on that one thread, which then permanently blocked every other test in the binary waiting on the same global lock. This explained the reported symptom precisely: hangs on every `cargo test` invocation (`test_gradient_check_all_ops` always runs), with the *apparent* hung test varying by scheduling (whichever tests happened to be mid-flight when the deadlocked thread never released the lock). Fixed by splitting `run_metal_src` into a lock-free `run_metal_src_locked` plus a thin locking wrapper, and having `test_gradient_check_all_ops` call the lock-free version from within its own single guard. Confirmed via `sample` on the hung process: every test thread's stack bottomed out in `Mutex::lock` with none actively executing, for the entire sampling window — the signature of an orphaned/self-held lock, not ordinary contention.

## Why this is surprising

Making `MetalContext` fields individually `Mutex`-guarded (see `metal.rs:76-77`'s "Metal objects are thread-safe... already serialized through the Mutex") reads as if the runtime is safe under concurrent use. It isn't: the mutexes prevent data races on individual fields, but not the logical protocol race across a full sequence of calls (compile → dispatch → barrier) that one logical "test" or "program run" makes. A future reader hitting a flaky hang might reach for full concurrency-safety (per-thread command buffers, namespaced pipeline caches) as the fix. That is a larger, deliberately deferred undertaking — see Considered alternatives.

## Considered alternatives

**Make MetalContext fully concurrency-safe.** Give every caller its own command buffer, namespace the pipeline cache per compilation, etc. Rejected for now: nothing in the M1–M29 roadmap gives the language a host-side concurrency feature (the language's actual concurrency model is GPU-side thread hierarchy within a kernel dispatch, shipped in M24, and is unaffected by this decision either way). Real host-thread concurrency would also require revisiting the autograd tape and CTMM/RC invariants, neither of which have ever been reasoned about under concurrent access (ADR-0015 already commits the tape to non-reentrant). Doing this as a side effect of a test-flakiness fix would be scope creep; it deserves its own milestone and ADR if the language ever grows a host-concurrency feature.

**Add a timeout to `gpu_barrier()`'s `waitUntilCompleted()`.** Considered as a way to turn future hangs into fast, diagnosable failures. Rejected: any fixed deadline would risk aborting legitimately long-running production GPU work (e.g. a long training loop). A test-only bounded wait was also considered and rejected — this fix is scoped to eliminating the known race, not general hang-diagnostics hardening.

## Consequences

- `cargo test --workspace` remains correct only because Metal-touching tests are serialized by convention (a `Mutex<()>` test lock duplicated in each of the two test files that touch real Metal, not shared cross-crate — the locks don't need to coordinate with each other since they guard separate OS processes). Any new test file added under `malus-runtime` or elsewhere that calls into the real Metal backend must adopt the same pattern, or it will reintroduce the hang.
- `MetalContext` remains unsafe to use from two host threads concurrently in production code too, not just in tests. This is consistent with, and should be read alongside, ADR-0015's "`backward` is not re-entrant."
- The test lock is a plain `std::sync::Mutex`, not reentrant. A test (or a helper it calls) must acquire it at most once per thread; if a helper function already takes the lock, callers that also need to take it must call the helper's lock-free inner variant instead of nesting acquisitions. `metal_integration.rs`'s `run_metal_src`/`run_metal_src_locked` split is the pattern to follow.
- `malus-codegen-gpu`'s own unit tests (`malus-codegen-gpu/src/tests.rs`) don't touch real Metal and aren't lock-serialized, but they do now share the process-global kernel-id counter across cargo's parallel test threads. Assertions in that file may only check relative id ordering (e.g. `sub_id > add_id`) within one compilation, never exact values or adjacency (e.g. never `sub_id == add_id + 1` or a hardcoded `Uniforms_0` in generated MSL) — a concurrently-running unrelated test can steal ids in between.
