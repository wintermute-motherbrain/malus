# CTMM: compile-time memory management with RC fallback

**⚠ Status: Superseded by ADR-0026 (V4).**  
The V1–V3 implementation shipped conservative lexical last-use drop insertion and type-directed RC on `Variable` (ADR-0016). The ADR-0002 "structural ambiguity → RC fallback" was never built. ADR-0026 implements the real Lobster-style borrow-inference analysis. ADR-0030 eliminates `Variable` and replaces type-directed RC with static grad-inference.


CTMM (Compile-Time Memory Management) uses escape analysis to insert static `free` calls for tensors at compile time, falling back to reference counting only when lifetimes are structurally ambiguous (tensors stored in heap containers or captured by closures). The RC fallback triggers for cold-path model parameter storage, not for hot-path tensor math.

The approach is inspired by the Lobster programming language's ownership model, adapted for a GPU-oriented tensor DSL.

## Considered Options

- **Garbage collection**: Unpredictable GC pauses are unacceptable for GPU workloads where the CPU must stay in sync with the Metal command queue.
- **Pure RC**: Correct and simple, but incurs unnecessary atomic increment/decrement overhead on every linear tensor flow — which is the hot path.
- **Rust-style ownership**: The borrow checker is too steep a learning curve for ML researchers who expect Python ergonomics. CTMM gives safety without requiring the user to think in lifetimes.
