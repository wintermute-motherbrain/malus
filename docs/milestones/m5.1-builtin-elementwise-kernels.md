# M5.1 ‚Äî Built-in Element-wise Kernels for fn-body BinOp

**Crate:** `malus-codegen-gpu` (synthesize built-in MSL); `malus-codegen-cpu` (lower tensor BinOp)
**Done when:** `fn add(a: Tensor<f32>, b: Tensor<f32>) -> Tensor<f32> { return a + b }` compiles and runs correctly, dispatching to a built-in `malus_add` kernel via `kernel_dispatch`.

## Background

ADR-0007 deferred tensor BinOp in `fn` bodies to a follow-up after M5. M5 delivered the `KernelRegistry`, `kernel_dispatch` ABI (`kernel_id: u64`), and real GPU dispatch. M5.1 uses this infrastructure to resolve ADR-0007.

## Scope

### Built-in kernel synthesis (`malus-codegen-gpu`)

`compile_kernels` synthesizes MSL for built-in element-wise operators when the program contains tensor BinOps in `fn` bodies. These built-in kernels are registered in the `KernelRegistry` alongside user-defined kernels, using sequential kernel_ids appended after user kernels (`N, N+1, ‚Ä¶`). The runtime is id-agnostic ‚Äî `runtime_init` derives the MSL function name as `malus_kernel_{id}` uniformly ‚Äî so built-ins are indistinguishable from user kernels below codegen-gpu.

Built-in kernels needed:
- `malus_add(lhs, rhs) -> out` ‚Äî element-wise `+`
- `malus_sub(lhs, rhs) -> out` ‚Äî element-wise `-`
- `malus_mul(lhs, rhs) -> out` ‚Äî element-wise `*`
- `malus_div(lhs, rhs) -> out` ‚Äî element-wise `/`

### Codegen-cpu lowering (`malus-codegen-cpu`)

When `lower_expr` encounters a `BinOp` on `ResolvedTy::Tensor` in a `fn` body, it lowers to a `kernel_dispatch` call:
1. Look up the built-in kernel's id from the name‚Üíid map
2. Stack-alloc the handles array (lhs, rhs)
3. Call `kernel_dispatch(builtin_id, handles, 2)`
4. Return the output handle

This replaces the current `UnsupportedExpr` error for tensor BinOp in `fn` bodies.

### CTMM interaction

The output of a built-in kernel dispatch is a pending tensor, same as a user kernel. CTMM's `extract_gpu_producing_expr` was extended in M5.1 to recognize tensor `BinOp` (and `Call` returning a GPU-placed tensor) in addition to `KernelCall`, so the existing barrier-insertion loop fires correctly. M5.1 also adds a `hoist_gpu_producing_returns` pass that desugars `return a + b` into `let __ret = a + b; return __ret;` before barrier insertion, ensuring the barrier fires *after* the dispatch (not before).

## Out of scope for M5.1

- Comparison ops on tensors (`==`, `<`, etc.) ‚Äî produce bool tensors, different lowering
- Unary ops on tensors in `fn` bodies (`-a`)
- Matmul in `fn` bodies ‚Äî MPS dispatch, not element-wise
- Fusion of `fn`-body BinOp chains into a single kernel
- Intermediate temporaries from chained BinOps (e.g. `b * c` in `a + b * c`) are not bound to names and therefore not tracked by CTMM's drop insertion. They leak. This is pre-existing behavior (nested user kernel calls leak identically) and is tracked in `docs/milestones/ctmm-v1-gaps.md`.

## Tests

- `fn add(a, b) -> Tensor<f32> { return a + b }` with two input tensors produces correct sum
- Chained BinOps: `fn fma(a, b, c) { return a + b * c }` dispatches two kernels
- Mixed user kernels and built-in BinOps in the same program
