# M5.1 — Built-in Element-wise Kernels for fn-body BinOp

**Crate:** `malus-codegen-gpu` (synthesize built-in MSL); `malus-codegen-cpu` (lower tensor BinOp)
**Done when:** `fn add(a: Tensor<f32>, b: Tensor<f32>) -> Tensor<f32> { return a + b }` compiles and runs correctly, dispatching to a built-in `malus_add` kernel via `kernel_dispatch`.

## Background

ADR-0007 deferred tensor BinOp in `fn` bodies to a follow-up after M5. M5 delivered the `KernelRegistry`, `kernel_dispatch` ABI (`kernel_id: u64`), and real GPU dispatch. M5.1 uses this infrastructure to resolve ADR-0007.

## Scope

### Built-in kernel synthesis (`malus-codegen-gpu`)

`compile_kernels` synthesizes MSL for built-in element-wise operators when the program contains tensor BinOps in `fn` bodies. These built-in kernels are registered in the `KernelRegistry` alongside user-defined kernels, using reserved kernel_ids (e.g., starting at `u64::MAX` and counting down, or a separate id space).

Built-in kernels needed:
- `malus_add(lhs, rhs) -> out` — element-wise `+`
- `malus_sub(lhs, rhs) -> out` — element-wise `-`
- `malus_mul(lhs, rhs) -> out` — element-wise `*`
- `malus_div(lhs, rhs) -> out` — element-wise `/`

### Codegen-cpu lowering (`malus-codegen-cpu`)

When `lower_expr` encounters a `BinOp` on `ResolvedTy::Tensor` in a `fn` body, it lowers to a `kernel_dispatch` call:
1. Look up the built-in kernel's id from the name→id map
2. Stack-alloc the handles array (lhs, rhs)
3. Call `kernel_dispatch(builtin_id, handles, 2)`
4. Return the output handle

This replaces the current `UnsupportedExpr` error for tensor BinOp in `fn` bodies.

### CTMM interaction

The output of a built-in kernel dispatch is a pending tensor, same as a user kernel. The existing CTMM pending-set logic handles barriers correctly without modification.

## Out of scope for M5.1

- Comparison ops on tensors (`==`, `<`, etc.) — produce bool tensors, different lowering
- Unary ops on tensors in `fn` bodies (`-a`)
- Matmul in `fn` bodies — MPS dispatch, not element-wise
- Fusion of `fn`-body BinOp chains into a single kernel

## Tests

- `fn add(a, b) -> Tensor<f32> { return a + b }` with two input tensors produces correct sum
- Chained BinOps: `fn fma(a, b, c) { return a + b * c }` dispatches two kernels
- Mixed user kernels and built-in BinOps in the same program
